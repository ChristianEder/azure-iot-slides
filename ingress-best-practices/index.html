Comparing Functions & Service Fabric

Experiences from a load test in a real world project

3000 simulated devices sent a total of 1.200 messages per minute over several hours to an IoT Hub. Two different architectures where evaluated
  • Event Processor Host and Service Fabric Actors based solution running on a cluster of 3 A2 machines.
  • Azure Function Event Hub trigger based solution running on a consumption plan
There where no complex requirements regarding data aggregation or analyzing the event stream, but the messages where serialized using Protobuf, which ruled out Stream Analytics

Comparing Functions & Service Fabric

Experiences from a load test in a real world project

The Functions based approach turned out to be...
  • faster - processing a message from IoT Hub into the storage took only ~200ms (compared to ~500ms-700ms in the SF approach)
  • cheaper to run - using Service Fabric, even for development environments you'd need a cluster of several machines, while the Functions consumption plan could handle the production load nearly for free (sorry, Microsoft ðŸ™‚)
  • easier to develop - being able to use existing components such as the Function Event Hub bindings saved us a ton of boilerplate code

... faster, cheaper, easier - this sounds too good to be true, right?
